# プロンプトの型の作り方

## 「やること」「やらないこと」はどこを見て書くか

3つのメモを順番に見ていくだけで埋まる。

---

## 手順

### Step 1. フォルダ構成.md → ファイルの一行説明を確認

そのファイルの横のコメントが「やること」の出発点になる。

```
AppStore.js  # グローバル状態（BPM、再生状態、マッピング等）
```

→「状態を持つファイルだ」とわかる。

---

### Step 2. 依存関係.md → そのファイルの項目を見る

依存関係.md には各ファイルに「持つもの」「依存先」が書いてある。
AppStore.js の場合：

```
持つ状態：
  bpm / isPlaying / x2 / selectedFile / effectStates / midiMappings
依存先：なし
```

→ 「やること」はここに書いてある状態を定義すること、とわかる。

---

### Step 3. 設計2.md → 仕様を確認して肉付けする

設計書に書かれた具体的な数値や条件を「やること」に追加する。

```
BPMは60〜200まで
x2機能あり
エフェクトは5種類（ピッチドロップ、グラニュラーフリーズ、スタッター、スクラッチ、シーケンサーゲート）
```

→「やること」の細かい中身が決まる。

---

### Step 4. 「やらないこと」は「依存している上位ファイルの仕事」を書く

依存関係.md で、AppStore.js を**使う側（上位）**のファイルを確認する。

```
AppStore.js を使っている：
  BpmSync.js / GridPlayer.js / EffectPanel.js / MidiMapping.js / FileSelector.js
```

これらのファイルがやること＝AppStore.js が**やってはいけないこと**。

例：
- BpmSync.js がBPMを計算する → AppStore.js はBPMを計算しない（ただ保存するだけ）
- EffectPanel.js がエフェクトのUI操作をする → AppStore.js はUIを作らない
- MidiMapping.js がMIDI割り当てを処理する → AppStore.js はMIDI処理をしない

---

## AppStore.js の場合、実際に埋めると

```
【作るファイル】
renderer/store/AppStore.js

【参照してよいファイル】
memo/設計2.md（状態の種類・仕様確認用）

【やること】
・以下の状態を定義して保持する
  - bpm（数値、初期値120、範囲60〜200）
  - isPlaying（真偽値）
  - x2（真偽値）
  - selectedFile（文字列、選択中のファイル名）
  - effectStates（5種類それぞれのオン/オフとパラメータ）
  - midiMappings（MIDI割り当て設定）
・それぞれの状態を読み書きする関数（getterとsetter）を用意する
・他のファイルが import して使えるようにエクスポートする

【やらないこと】
・BPMの計算や同期処理（→ BpmSync.js の仕事）
・エフェクトの音処理（→ effects/ の仕事）
・MIDI信号の受信・処理（→ MidiController.js / MidiMapping.js の仕事）
・UIの描画（→ ui/ の仕事）
・ファイルの読み込み（→ SliceManager.js の仕事）
```

---

## まとめ：3つのメモとの対応

| 項目 | 見るメモ | 何がわかるか |
|---|---|---|
| 作るファイル | フォルダ構成.md | ファイルの一行説明 |
| 参照ファイル | 依存関係.md（依存先の欄） | 読んでいいファイル |
| やること | 依存関係.md（持つもの）＋設計2.md | 状態・処理の具体的な中身 |
| やらないこと | 依存関係.md（使う側のファイル） | 隣のファイルに任せる処理 |

---

## 迷ったときの判断基準

「これを AppStore.js に書くべきか？」と迷ったとき：

→ **「他のファイルが import して使うデータか？」**
　 YES → AppStore.js に書く（状態の定義）
　 NO  → 別のファイルの仕事（処理・計算・描画）
