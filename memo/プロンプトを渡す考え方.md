# Claude Code へのプロンプトの渡し方

## 基本方針

### 1. 会話は1ファイル（または密接な2ファイル）で完結させる
Claude Code は会話が長くなるほど、過去のやり取りを参照しながら動くため、
**1つの会話 = 1つのファイルを完成させる** を原則にする。

完成したら会話を閉じ、次のファイルは新しい会話で始める。

---

### 2. 作る順番は「依存関係.md の下から上へ」

依存関係.md の図で、**矢印の先（依存される側）から先に作る**。
依存される側が存在しないと、Claude が「何に繋げばいいかわからない」状態になる。

```
作る順番（上が先）

1. store/AppStore.js          ← 依存なし・最初に作る
2. audio/AudioEngine.js       ← 依存なし・最初に作る
3. audio/SliceManager.js      ← AudioEngine に依存
4. audio/BpmSync.js           ← AudioEngine に依存
5. audio/TimeStretcher.js     ← AudioEngine に依存
6. audio/FadeManager.js       ← AudioEngine に依存
7. effects/*.js（個別）       ← AudioEngine に依存
8. effects/EffectChain.js     ← effects/* + AudioEngine に依存
9. audio/GridPlayer.js        ← 上記すべてに依存・最後のほうに作る
10. midi/MidiController.js    ← IPC のみ
11. midi/MidiMapping.js       ← MidiController + AppStore に依存
12. ui/FileSelector.js        ← SliceManager + AppStore に依存
13. ui/GridView.js            ← GridPlayer + AppStore に依存
14. ui/WordDisplay.js         ← SliceManager + GridPlayer に依存
15. ui/EffectVisualizer.js    ← SliceManager + GridPlayer に依存
16. ui/EffectPanel.js         ← EffectChain + AppStore に依存
17. main/midi/midiMain.js     ← node-midi + IPC
18. main/link/linkBridge.js   ← abletonlink + IPC
19. main/main.js              ← すべてのメインプロセス側を統合
20. renderer/index.js         ← すべてのレンダラー側を統合
```

---

### 3. プロンプトの型

毎回この形で伝えるとトークンの無駄が減り、精度が上がる。

```
【作るファイル】
renderer/audio/SliceManager.js

【参照してよいファイル】
renderer/audio/AudioEngine.js（既存）
memo/設計2.md（仕様確認用）

【やること】
・AudioEngine の AudioContext を使い、
  assets/json/XXX.json を読み込んでモーラ単位にスライスする
・スライスは { buffer, word, mora, startTime, rms, f0, spectral_centroid, zcr } の配列で持つ

【やらないこと】
・他のファイルは読まない
・UIは作らない
・テストは書かない
```

**ポイント：「参照してよいファイル」を明示する**
これを書かないと Claude は勝手にいろんなファイルを読みに行く。
依存関係.md を見て、そのファイルが依存しているもの「だけ」を渡す。

---

### 4. 渡すファイルの判断方法（依存関係.md の使い方）

依存関係.md の各ファイルの表を見る。

例：`GridPlayer.js` を作りたい場合

```
audio/GridPlayer.js の依存先：
  - audio/AudioEngine.js
  - audio/SliceManager.js
  - audio/TimeStretcher.js
  - audio/BpmSync.js
  - audio/FadeManager.js
  - store/AppStore.js
```

→ プロンプトの「参照してよいファイル」にこの6つを書く。
　それ以外は書かない（読まれてもトークンの無駄）。

---

### 5. IPC をまたぐファイルはセットで作る

メインプロセスとレンダラープロセスは IPC で繋がっている。
片方だけ作ると動作確認できないので、以下はセットで作る。

| メイン側 | レンダラー側 |
|---|---|
| `main/link/linkBridge.js` | `audio/BpmSync.js` |
| `main/midi/midiMain.js` | `midi/MidiController.js` |
| `main/main.js`（asset一覧） | `ui/FileSelector.js` |

セットで作るときは同じ会話内で両方指示する。
ただし1ファイル完成 → 次のファイル、と順番に進める。

---

### 6. 動作確認のタイミング

以下のタイミングで「ここまで動くか確認してください」と伝える。

- `AudioEngine.js` 完成後 → `AudioContext` が起動するか
- `SliceManager.js` 完成後 → スライス配列が正しく生成されるか
- `GridPlayer.js` 完成後 → 音が出るか
- `EffectChain.js` 完成後 → エフェクトが切り替わるか

この確認を省くと、後で原因特定が難しくなる。

---

### 7. 修正・バグ対応のとき

バグが出たら、**依存関係.md の「バグを追うときの手順」** を参照して、
疑わしいファイルを1〜2個に絞ってから指示する。

```
例：
「音が出ません。AudioEngine.js と GridPlayer.js だけを見て、
　AudioContext が running 状態かどうかと、
　バッファの再生呼び出しを確認してください」
```

複数ファイルにまたがる可能性があっても、まず1ファイルに絞る。
直らなければ次のファイルへ移る。

---

### まとめ：指示の鉄則

| やること | やらないこと |
|---|---|
| 1会話 = 1ファイル | 「全部作って」はNG |
| 参照ファイルを明示する | ファイル指定なしで渡す |
| 依存の下から順に作る | 依存先がない状態で上位を作る |
| 動作確認を挟む | 全部作ってから確認する |
| バグは1〜2ファイルに絞る | 「全体を見て直して」はNG |
